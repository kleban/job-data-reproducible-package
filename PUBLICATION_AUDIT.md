# Pre-Publication Audit for Mendeley Data

**Repository:** *Labor Demand for Digital Skills in Post-2022 Ukraine: Evidence from Online Job Vacancy Data*  
**Authors:** Yurii Kleban and Britta Rude  
**Audit date:** 2026-07-21  
**Scope:** the 248 tracked or unignored files currently proposed for publication, totaling 453,360,495 bytes (432.36 MiB)

## Decision

**The repository is not ready to publish to Mendeley Data.** Its organization and internal path hygiene are strong, but legal clearance and final computational verification remain incomplete. The two ACLED source workbooks must not be included in an open deposit under the currently published ACLED terms unless ACLED provides written permission. The Jooble-derived data also require written confirmation of the authors' right to redistribute each derived file.

The package is well below Mendeley Data's current 10 GB per-dataset limit. The complete candidate-file register, sizes, checksums, roles, and proposed release actions are in [FILE_INVENTORY.md](FILE_INVENTORY.md).

This audit is a technical and open-science review, not legal advice. Contractual and copyright decisions should be confirmed by the authors' institutions or qualified counsel.

## Research-description audit

The manuscript and root README agree on the title, author order, and broad purpose. The Mendeley landing-page description should be expanded to include the actual study scope and design:

> Replication materials for “Labor Demand for Digital Skills in Post-2022 Ukraine: Evidence from Online Job Vacancy Data” by Yurii Kleban and Britta Rude. The study analyzes 17.1 million Jooble online job vacancies in Ukraine from January 2021 through June 2025. Vacancy text is processed with large language models and mapped to ESCO; weekly and monthly labor-demand measures are linked to ACLED conflict events and fatalities. Time-series analyses document a positive association—not a causal effect—between conflict intensity and the share of vacancies requiring digital skills. The paper examines within- and between-occupation changes, remote work as a mediating channel, and the relative resilience of digital hiring during a contraction in total vacancy volume.

Add the following metadata to the root README and Mendeley record after author confirmation:

- author affiliations and ORCID identifiers;
- keywords: Labor Demand; Technological Change; Large Datasets; National Security; War; Conflict;
- JEL codes: J23, O33, C55, H56;
- data coverage: Ukraine, January 2021–June 2025;
- the article DOI when assigned and the final Mendeley Data DOI/version;
- funding, acknowledgements, and any institutional disclaimer required by the World Bank or the authors' universities;
- an explicit statement that the time-series findings are associative rather than causal.

## README and documentation findings

### Root README

The root README has a useful component map, data-availability table, environment split, and four reproduction routes. Before publication it should be revised as follows:

1. Replace statements that ACLED files are included and publicly redistributable with access instructions and a description of permissible aggregated derivatives.
2. State which route is the authoritative route for reproducing the paper. At present Route C cannot run from the public files, while Route D has not been tested.
3. Add a complete output crosswalk using the manuscript's visible labels and numbers, not only internal LaTeX labels.
4. Add expected runtime, memory, internet access, API cost, nondeterminism, and operating-system notes for each route.
5. Separate “can reproduce the paper” from “demonstrates the data schema.” The synthetic pipeline cannot reproduce substantive estimates.
6. Remove links to internal progress files from the release README after their outstanding items are moved into a public limitations section or resolved.
7. Add a component-level licensing table; one Mendeley license cannot override third-party or restricted-data terms.

## Code and environment findings

### Python

- `requirements.txt` covers the imports found by static analysis, but the package has not been recreated from a clean environment.
- The documented Python version is 3.13.12; the local interpreter observed during this audit was different, and the existing local virtual environment has a broken `pyarrow` binary installation. This is evidence that a clean install test is still required.
- Comments in `requirements.txt` describe some pins as “current stable”; all pins should instead come from the successfully tested release environment.
- `before_start_test_environment.ipynb` should report only whether `OPENAI_API_KEY` exists. It currently contains code that would print the first eight characters of a live key at runtime.
- Fourteen `.pkl` files are Python-specific and can execute code when unpickled. Publish an open, non-executable equivalent such as Parquet/CSV/JSON plus a schema wherever practical; retain Pickle only when a documented consumer requires it.

### R

- `DESCRIPTION` has the wrong paper title, reverses the manuscript author order, declares `License: MIT` without a license file, and does not directly declare all packages referenced with namespace syntax (`dplyr`, `tidyr`, `scales`, and `renv`).
- `run_all.R` catches an error, continues to later scripts, and always prints “Pipeline complete.” This can create a false successful run. It should fail fast or finish with a nonzero status and a list of failed scripts.
- Several R scripts install missing packages at runtime. That behavior weakens the locked `renv` environment; scripts should stop with restoration instructions instead.
- `R/13_baseline_regression.R` is a near-duplicate of Script 02 and is not called by `run_all.R`. Remove it from the deposit or document exactly why both versions are necessary.
- `R/12_mechanisms_remoteshare.R` is described in `run_all.R` as “causal mediation,” while the manuscript frames the evidence associatively. Align code comments and documentation with the paper's identification claims.

## Paths, filenames, and duplication

The active code and documentation contain no machine-specific absolute paths, and all tested local links resolve. The remaining issues are semantic rather than path-resolution failures:

- Historical notebook names contain misspellings or misleading stage numbers: `stage_01_5_interim_translate_skills.ipynb`, `stage_2_1_skills_extration.ipynb`, and `stage_2_2_add_romote_jobs.ipynb`; `unic_id_db.pkl` is also misspelled. Either rename them once with every reference updated and retested, or add a prominent canonical-name alias table. Do not rename files after the final output crosswalk is frozen.
- Identical ESCO files are repeated: `skills_en.csv` appears three times and `skills_ru.csv` twice. Prefer one canonical copy plus documented consumers, unless keeping local copies is necessary for a self-contained component.
- The synthetic Stage 5 daily “full” and “unique” Parquet files are byte-identical, as are corresponding JSON demonstration artifacts. Verify that this is an intentional consequence of the synthetic sample and state it explicitly; otherwise it can look like a filtering error.
- `region_db.pkl`, `region_db_NEW.pkl`, and `region_db_PREV.pkl` do not reveal which is authoritative. Retain one canonical lookup and document provenance, or move non-authoritative versions out of the deposit.
- The raw ACLED workbook is duplicated solely for R-project autonomy. Once the prohibited source copies are removed, both Python and R should consume a documented permissible aggregate or instruct the replicator how to obtain ACLED independently.
- The package contains five exact duplicate-content groups. Deduplication is not required for the 10 GB size limit, but it reduces ambiguity and makes checksums and legal scope easier to manage.

## Data, privacy, and disclosure review

### Jooble-derived data

The original vacancy snapshots are appropriately excluded and the README states that they are restricted. However, “derived” does not automatically mean redistributable. Before release, create a signed data-release decision for each public Jooble-derived file class:

- R analysis-ready Parquet panels;
- reviewer-validation population, sample, template, and manual codes;
- Stage 1–5 synthetic demonstration intermediates;
- OpenAI Batch inputs and outputs;
- lookup tables and record identifiers.

The reviewer template contains `id`, `clean_title`, and `clean_desc`; even when no obvious email address or URL is detected, titles and descriptions can contain personal or employer information and may reproduce protected vacancy text. A human disclosure review is required. Apply data minimization, pseudonymize identifiers, suppress unnecessary free text, and document the lawful basis and safeguards.

The static text scan found no email addresses or web URLs in the candidate reviewer/Batch text files. A broad phone-number pattern produced many matches in Batch JSONL, but inspection indicates that the measure is dominated by long numeric API/request identifiers; it is not a reliable privacy clearance. Parquet contents could not be scanned because the current local `pyarrow` installation is broken.

### OpenAI Batch material

Batch output ownership under OpenAI's service agreement does not grant rights in third-party input data. OpenAI's current agreement makes the customer responsible for having all rights and permissions for inputs and for evaluating outputs. Retain prompts and deterministic post-processing code where allowed, but minimize platform metadata and do not publish Batch inputs/outputs until Jooble rights, personal-data review, and Mendeley admissibility are resolved.

### Data-protection principles

Vacancy text can contain names, emails, telephone numbers, employer contacts, and other identifiers. The GDPR principles relevant to publication include lawfulness, purpose limitation, data minimization, accuracy, storage limitation, integrity/confidentiality, and accountability. Research use does not automatically authorize unrestricted public disclosure. Keep only fields necessary for independent verification and document anonymization or aggregation decisions.

## Copyright and licensing review

| Material | Proposed action |
|---|---|
| Author-written Python/R code and documentation | Select an author-approved open-source license, add the full license text, copyright holders, and year, and make `DESCRIPTION` consistent. Do not assume the authors personally own employer-created code without checking institutional policy. |
| Author-created data/documentation | Use a separate data/documentation license only for material the authors are authorized to license. CC BY 4.0 is a reasonable open-data option when ownership is clear, but it cannot relicense restricted Jooble or ACLED material. |
| ACLED source workbooks | Exclude from public files unless ACLED grants written permission. Provide access/filter/date instructions and, if permitted, only sufficiently aggregated non-reconstructable derivatives with prominent attribution. |
| ESCO exports | Confirm the exact dataset release and official reuse notice. Preserve concept URIs, provenance, attribution, and any applicable EU reuse/EUPL notices; do not describe the API's EUPL license as automatically governing every downloaded taxonomy file without confirmation. |
| Electricity/energy workbooks | Confirm copyright/database rights, source terms, and permission to redistribute the compiled sheets; document every transformation and source URL/date. |
| Manuscript PDF | Mendeley describes research data rather than the narrative article and prohibits copyrighted content the depositor does not own. Omit the PDF unless the journal/publisher and all rightsholders permit this exact manuscript version to be redistributed; link the article DOI instead. |
| Prompts and model outputs | Confirm Jooble/input rights, OpenAI terms, Mendeley admissibility, and whether prompt text contains protected vacancy material. |

The Mendeley license selector defaults to CC BY 4.0. Do not accept the default for this mixed-rights package without a written component-by-component decision. If one repository record cannot express the needed rights cleanly, deposit only author-controlled code/data and provide access instructions or metadata for restricted sources.

## Open-science and FAIR improvements

1. Add `CITATION.cff` with the paper title, authors, ORCIDs, repository version, related article identifier, and later the Mendeley DOI.
2. Add a root `LICENSE` for code and a `LICENSES.md` or `THIRD_PARTY_NOTICES.md` matrix for datasets, taxonomies, model outputs, and the manuscript.
3. Add a machine-readable release manifest with path, byte size, SHA-256, role, source, license, and producer. `FILE_INVENTORY.md` is the human-readable starting point.
4. Add schema/codebook files for every public analysis dataset, including units, types, missing-value conventions, temporal coverage, geographic coverage, and transformations.
5. Record software and model provenance: Python/R versions, OS, package locks, `gpt-4o-mini` snapshot or dated model identifier if available, sentence-transformer model name/revision, seeds, Batch dates, and nondeterministic limitations.
6. Add a clean-environment validation log containing commands, timestamps, runtimes, peak memory, and checksums of generated outputs.
7. Include only manuscript-used outputs in the release, or place exploratory products in a clearly excluded/archive section.
8. Use open formats alongside proprietary formats: CSV/Parquet plus codebooks for Pickle/Excel where legally permitted. Mendeley recommends CSV/ODS for spreadsheet-style data and treats XLS/XLSX as acceptable rather than preferred formats.
9. Assign a semantic release version such as `1.0.0` and tag the exact deposited tree. Do not publish until the deposit archive checksum and inventory match the approved local release.
10. Add a data-availability statement that distinguishes restricted raw data, synthetic demonstration data, directly shared analysis-ready data, externally obtainable sources, and unavailable historical record-level threshold predictions.

## Recommended release sequence

1. Freeze the current repository as a private working version.
2. Obtain Jooble, ACLED, energy-source, institutional, and manuscript-rights decisions in writing.
3. Resolve Mendeley's AI-generated-data rule with the journal data editor or Mendeley support.
4. Create a clean release tree containing only approved files.
5. Repair metadata, licensing, pseudo-data, codebooks, and the complete manuscript crosswalk.
6. Run Python and R validation in clean environments; compare all retained outputs with the manuscript.
7. Regenerate the inventory and checksums from the release tree.
8. Upload as a Mendeley draft, add title, description, contributors/ORCIDs, category, collection dates, nature of data, license, related article, and exact steps to reproduce.
9. Preview and independently inspect the downloaded draft archive.
10. Publish the approved version, wait for moderation and the active DOI, then add that exact DOI/version to the journal's File Inventory as Research Data.
